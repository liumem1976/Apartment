# Contributing

感谢你对本项目的贡献！请在提交 PR 或 issue 前遵循下面的指南，以便我们更高效地协作。

1. 报告问题 / 提需求
   - 在提交 issue 时，说明问题或需求的复现步骤、期望行为以及实际行为。附上最小可复现示例（如果可能）。

2. 提交代码（Pull Request）
   - 在开始实现前，建议先创建 issue 讨论实现方案（小改动可以跳过）。
   - 新功能或修复请基于 `main` 分支新建 feature 分支，分支命名示例：`feat/<短描述>`、`fix/<短描述>`。
   - 每个 PR 应该包含清晰的标题和描述，说明改动的动机与影响。
   - 如果改动涉及数据库或迁移，请在 PR 描述中明确列出需要执行的迁移步骤。

3. 代码风格与质量
   - 使用现有的 linters/formatters：`ruff`, `black`, `isort`, `flake8`。
   - 在提交前运行 `pytest` 并确保测试通过。
   - 不要在 unrelated 文件中引入样式更改。

4. 提交信息
   - 使用简明的 commit message，首行为简要描述，后面可加空行并补充详细说明（必要时）。
   - 可采用 `type(scope): subject` 风格，例如 `fix(billing): correct arrears calculation`。

5. 测试
   - 为新功能或 bug 修复添加相应的测试用例。
   - CI 会在 PR 中运行测试，测试通过后才会合并。

6. 代码审查
   - 自动化检查通过后，请 @ 指定相关同伴进行 review。
   - 维护者可能会要求你对实现进行调整，请尽量配合修改。

7. 其他
   - 若对贡献流程有建议，请提出 issue 讨论。

谢谢！
